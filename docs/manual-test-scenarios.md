# 手動テストシナリオ

このドキュメントは、アプリケーションの手動テスト手順を定義します。

## テスト環境

- **ブラウザ**: Chrome, Firefox, Safari, Edge
- **デバイス**: デスクトップ, タブレット, モバイル
- **URL**: http://localhost:5173/Team-Assignment/ (開発) / https://ta-hasunuma.github.io/Team-Assignment/ (本番)

## シナリオ 1: 基本的なチーム分け

### 目的

最も基本的な使用フローでチーム分けが正常に動作することを確認

### 前提条件

- アプリケーションが起動している
- 初期状態（メンバー0人）

### テスト手順

1. **メンバー追加**
   - [ ] 「名前」フィールドに「太郎」と入力
   - [ ] 「NAiS」ラジオボタンが選択されていることを確認
   - [ ] 追加ボタン（+）をクリック
   - [ ] メンバーリストに「太郎」が表示されることを確認
   - [ ] 入力フィールドがクリアされることを確認

2. **複数メンバー追加**
   - [ ] 同様に「花子 (KAG)」「次郎 (NAiS)」「三郎 (KAG)」を追加
   - [ ] メンバー一覧に4名表示されることを確認
   - [ ] statsに「合計: 4, NAiS: 2, KAG: 2」が表示されることを確認

3. **チーム数設定**
   - [ ] 「全体のWIP数」スライダーを2に変更
   - [ ] バッジに「2」が表示されることを確認

4. **チーム作成**
   - [ ] 「チームを作成」ボタンをクリック
   - [ ] ローディングアニメーションが表示されることを確認
   - [ ] 結果セクションが表示されることを確認
   - [ ] 2つのチームカードが表示されることを確認
   - [ ] 各チームに2名ずつ配置されることを確認
   - [ ] heroセクションに「2つのチームが作成されました」が表示されることを確認

5. **結果クリア**
   - [ ] 「結果をクリア」ボタンをクリック
   - [ ] チーム結果が非表示になることを確認
   - [ ] メンバーリストは保持されていることを確認

### 期待結果

- すべてのステップがエラーなく完了する
- UIが適切に更新される
- チームが均等に分けられる

## シナリオ 2: パターン制約付きチーム分け

### 目的

パターンルール機能が正常に動作することを確認

### 前提条件

- メンバーが6名以上登録されている（NAiS: 3名以上, KAG: 3名以上）

### テスト手順

1. **NAiSのみチーム設定**
   - [ ] 「新しいルールを追加」セクションを展開
   - [ ] チームタイプで「🚀 NAiSのみ」ボタンをクリック
   - [ ] チーム数スライダーを1に設定
   - [ ] 1チームの人数スライダーを3に設定
   - [ ] 「ルールを追加」ボタンをクリック
   - [ ] ルールが一覧に追加されることを確認

2. **KAGのみチーム設定**
   - [ ] チームタイプで「🎯 KAGのみ」ボタンをクリック
   - [ ] チーム数を1、人数を3に設定
   - [ ] 「ルールを追加」ボタンをクリック

3. **全体チーム数設定**
   - [ ] 「全体のWIP数」を3に設定

4. **バリデーション確認**
   - [ ] エラーメッセージが表示されないことを確認
   - [ ] 警告メッセージが表示される場合、内容を確認

5. **チーム作成**
   - [ ] 「チームを作成」ボタンをクリック
   - [ ] 3つのチームが作成されることを確認
   - [ ] NAiSのみチームに3名のNAiSメンバーが配置されることを確認
   - [ ] KAGのみチームに3名のKAGメンバーが配置されることを確認
   - [ ] 残りのメンバーが3つ目のチームに配置されることを確認

6. **ルール削除**
   - [ ] 各ルールの削除ボタン（×）をクリック
   - [ ] ルールが削除されることを確認

### 期待結果

- パターンルールが正しく適用される
- 制約違反がある場合、適切なエラーが表示される
- 残りメンバーが適切に分配される

## シナリオ 3: バリデーションエラー

### 目的

バリデーション機能が適切にエラーを検出し表示することを確認

### テスト手順

1. **メンバー0人でのチーム作成**
   - [ ] メンバーを全削除（全削除ボタン）
   - [ ] 「チームを作成」ボタンが無効化されることを確認
   - [ ] エラーメッセージ「メンバーが1人以上必要です」が表示されることを確認

2. **チーム数超過**
   - [ ] メンバーを3名追加
   - [ ] チーム数を5に設定
   - [ ] エラーメッセージ「チーム数はメンバー数以下である必要があります」が表示されることを確認
   - [ ] 「チームを作成」ボタンが無効化されることを確認

3. **NAiSメンバー不足**
   - [ ] NAiSメンバーを2名、KAGメンバーを4名追加
   - [ ] パターンルールで「NAiSのみ 1チーム × 3名」を設定
   - [ ] エラーメッセージ「NAiSのみのチームを作成するにはNAiSメンバーが不足しています」が表示されることを確認

4. **KAGメンバー不足**
   - [ ] NAiSメンバーを4名、KAGメンバーを2名に変更
   - [ ] パターンルールで「KAGのみ 1チーム × 3名」を設定
   - [ ] エラーメッセージ「KAGのみのチームを作成するにはKAGメンバーが不足しています」が表示されることを確認

### 期待結果

- すべてのバリデーションエラーが適切に検出される
- エラーメッセージが明確で理解しやすい
- エラーがある場合、チーム作成ボタンが無効化される

## シナリオ 4: UI/UX 確認

### 目的

ユーザーインターフェースとユーザーエクスペリエンスの品質を確認

### テスト手順

1. **アニメーション**
   - [ ] ページ読み込み時のフェードインアニメーションを確認
   - [ ] メンバー追加時のアニメーションを確認
   - [ ] チーム作成時のローディングアニメーションを確認
   - [ ] 結果表示時のスライドインアニメーションを確認

2. **ホバーエフェクト**
   - [ ] ボタンホバー時の拡大・シャドウ効果を確認
   - [ ] カードホバー時のリフトアップ効果を確認
   - [ ] メンバーリストのテーブル行ホバー効果を確認

3. **レスポンシブデザイン**
   - [ ] デスクトップ表示（1920px）で2カラムレイアウトを確認
   - [ ] タブレット表示（768px）でレイアウトを確認
   - [ ] モバイル表示（375px）で1カラムレイアウトを確認
   - [ ] すべての要素が適切に収まることを確認

4. **アクセシビリティ**
   - [ ] Tabキーでフォーカス移動ができることを確認
   - [ ] Enterキーでボタンが押せることを確認
   - [ ] ラジオボタンが矢印キーで選択できることを確認
   - [ ] すべてのインタラクティブ要素にフォーカスリングが表示されることを確認

5. **カラーとコントラスト**
   - [ ] テキストが読みやすいコントラストであることを確認
   - [ ] プライマリカラー（青）が一貫して使用されることを確認
   - [ ] エラー（赤）、警告（黄）、成功（緑）の色分けを確認

### 期待結果

- すべてのアニメーションが滑らかに動作する
- ホバーエフェクトが直感的である
- レスポンシブデザインが適切に機能する
- アクセシビリティ基準を満たしている

## シナリオ 5: エッジケース

### 目的

極端な状況でもアプリケーションが安定して動作することを確認

### テスト手順

1. **大量メンバー**
   - [ ] 50名以上のメンバーを追加
   - [ ] メンバーリストがスクロール可能であることを確認
   - [ ] チーム作成が正常に動作することを確認
   - [ ] パフォーマンスが許容範囲内であることを確認

2. **最小構成**
   - [ ] メンバー1名、チーム数1でチーム作成
   - [ ] 正常に動作することを確認

3. **最大チーム数**
   - [ ] チーム数を20（最大値）に設定
   - [ ] スライダーが20以上にならないことを確認

4. **長い名前**
   - [ ] 50文字以上の長い名前を入力
   - [ ] UI が適切に表示されることを確認
   - [ ] テキストが適切に折り返されることを確認

5. **特殊文字**
   - [ ] 絵文字を含む名前を入力（例: 太郎 🚀）
   - [ ] 正常に表示されることを確認

6. **連続操作**
   - [ ] チーム作成を連続10回実行
   - [ ] メモリリークがないことを確認
   - [ ] 毎回ランダムな結果が得られることを確認

### 期待結果

- すべてのエッジケースで安定して動作する
- エラーが発生しない
- パフォーマンスが許容範囲内である
